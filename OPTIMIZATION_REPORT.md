### GUI Agent 性能优化方案与评估报告

#### 一、现状与问题分析

结合当前代码结构与运行路径，GUI Agent 的任务执行速度主要受以下几类因素影响：

- **远程多模态模型调用延迟**：
  - 每一步都通过灵芽兼容的 OpenAI Chat Completions 接口调用多模态模型，网络往返与推理时间是整个流程中最重的一环。

- **会话历史无限累积，且重复携带图片**（原始实现）：
  - `GeminiChat` 在 `get_multimodal_response` 中，将完整消息（包含 base64 图片）追加到 `conversation_history`，并在下一轮调用时整体发送。
  - 随着步数增加，请求体中会携带越来越多的 base64 图片与文本，导致：
    - 请求体变大，网络传输时间上升；
    - 客户端序列化/反序列化开销增大；
    - 服务端处理上下文的时间增加。

- **等待时间偏保守**：
  - 操作层（`gui_operator/execute.py`）中存在多处固定 `time.sleep`，以及 `wait()` 动作默认等待 5 秒。
  - 对于多数常见桌面操作，这明显偏长，会显著拉高端到端耗时。

- **缺少统一的性能调节入口与基准测试工具**：
  - 等待时间与模型返回长度等关键性能参数，难以在不同机器/网络环境下做细粒度调节和对比测试。

综合来看，当前性能瓶颈主要集中在：**多模态模型请求的体积与频率**，以及 **保守的等待时间配置**，算法复杂度和数据结构本身并不是核心问题。

---

#### 二、代码级优化措施

##### 1. 精简与限制会话历史（`utils/model.py`）

- **目标**：减少每次模型调用的上下文体积，尤其是避免重复传输大量 base64 图片。
- **具体改动**：
  - 引入 `max_history_turns` 与 `max_tokens` 配置：
    - `max_history_turns`：限制仅保留最近 N 轮对话（user+assistant 为一轮），防止历史无限增长。
    - `max_tokens`：控制单次返回 token 数量，降低推理与传输时间，默认值从 2048 降到 512。
  - 历史仅保存「纯文本对话」，不再保存图片：
    - 当前轮仍发送一张最新截图（`image_url` + `text`）。
    - 历史只记录此前的 user 文本指令与 assistant 文本回复。
    - 这样每步只会附带一张图片，避免图片历史成倍叠加。

**效果预期**：

- 对于步数较多的任务，请求体大小将显著减小，上下文构建与网络传输时间都会下降。
- 在保证一定上下文记忆能力的前提下，整体吞吐与响应速度提升明显。

##### 2. 等待时间的统一倍率与精细控制（`gui_operator/execute.py`）

- **目标**：在不同环境下灵活调节操作等待时间，实现速度/稳定性的可控权衡。
- **具体改动**：
  - 新增全局环境变量 `GUI_AGENT_SPEED`：
    - `GUI_AGENT_SPEED=0.5`：整体等待时间减半（更快，适合本机响应较快的场景）。
    - `GUI_AGENT_SPEED=1.0`：保持原有等待策略。
    - `GUI_AGENT_SPEED=2.0`：整体等待时间翻倍（更稳但更慢）。
  - `Operation.wait` 改为：根据 `GUI_AGENT_SPEED` 计算 `effective` 等待时间，并设置下限 `0.05s`，避免过短导致不稳定。

**效果预期**：

- 在不改业务逻辑的前提下，通过简单调节环境变量即可全局加速或放缓 GUI 操作，利于在不同机器上做性能试验与调优。

##### 3. 缩短 `wait()` 动作的默认等待时长（`main.py`）

- **目标**：减少模型输出 `wait()` 动作时对整体流程的阻塞时间。
- **具体改动**：
  - 将 `_parse_and_execute` 中 `wait()` 对应的等待时间从 **5 秒** 调整为 **2 秒**：
    - 同时仍然会经过 `Operation.wait`，受 `GUI_AGENT_SPEED` 影响。

**效果预期**：

- 在常见桌面应用响应时间通常低于数秒的情况下，2 秒 + 倍率调整可以在「稳定」与「速度」之间取得更好的平衡。

#### 三、架构与资源配置层面的建议

##### 1. 模型调用侧

- **减少不必要的调用次数**：
  - 如果任务本身对精细交互要求不高，可以在 Prompt 设计上让模型在每步完成更多子任务，从而减少总步数（`max_steps` 可适当降低）。
- **选择更快的模型/路由**：
  - 当前默认模型为 `gemini-3-flash-preview`，已经是偏快速的配置，如网络环境允许，可保留此配置以获得更好吞吐。

##### 2. GUI 层与系统资源

- **避免后台程序大量占用 CPU/GPU**：
  - 由于截图与 GUI 操作依赖系统资源，建议在进行大量 Agent 任务测试时，减少其他高占用进程。
- **多 Agent 并行时的资源规划**：
  - 如在未来希望并行运行多个 GUI Agent，应通过进程级并发（多进程）而非多线程，以避免 GIL 对 Python 端 CPU 密集任务的影响，并注意避免多个实例同时抢占鼠标键盘控制权。

#### 四、基准测试设计与使用方式

为验证优化效果，新增脚本 `benchmark_agent.py`，用于做端到端性能对比。

##### 1. 脚本说明

- 入口函数：`benchmark_run(instruction: str, repeat: int = 1, model_name: Optional[str] = None)`
- 功能：
  - 重复多次创建 `GUIAgent` 并执行 `run()`；
  - 记录每次完整任务的耗时，并输出平均值。

##### 2. 推荐对比方法

1. **优化前基线（如使用旧版本代码时）**：
   - 在同一台机器上，选择一条代表性指令，例如：
     - “打开浏览器搜索某条信息，然后结束任务。”
   - 设置：`GUI_AGENT_SPEED=1.0`（或不设环境变量，使用默认）。
   - 运行脚本，记录平均耗时。

2. **启用优化后的对比**：
   - 使用当前优化后的代码，保持其它条件不变：
     - 相同指令、相同网络环境、相同机器。
   - 运行：
     - 默认速度（`GUI_AGENT_SPEED=1.0`）；
     - 加速场景（如 `GUI_AGENT_SPEED=0.5`）。
   - 对比平均耗时与任务完成情况是否保持稳定。

> 由于远程模型与网络环境存在波动，建议每组对比至少执行 3~5 次，取平均值进行评估。

#### 五、测试结果解读建议

在实际落地时，可按以下维度评估优化效果：

- **端到端平均耗时**：
  - 关注「从 Agent 启动到任务完成」的总时间变化。
  - 在多数场景下，预期可在保持任务成功率不明显下降的前提下，有 10%~30% 的提速空间（视网络与任务复杂度而定）。

- **单步模型响应时间**：
  - 可以在日志中观察每次调用 `get_multimodal_response` 前后的打印时间差，验证 `max_tokens` 与历史精简带来的影响。

- **任务完成准确性与稳定性**：
  - 观察是否出现因为等待过短导致的界面尚未更新就继续操作的问题；
  - 如若出现，可适当提高 `GUI_AGENT_SPEED` 或将部分关键操作后的等待时间调高。

---

#### 六、后续性能监控与进一步优化方向

- **增加细粒度时间日志**：
  - 在关键节点（截图、模型调用前后、动作执行前后）打印时间戳或耗时，用于精确定位瓶颈。

- **更精细的历史控制策略**：
  - 当前实现按轮数限制历史，后续可进一步按「估算 token 数」进行裁剪，以便在复杂文本场景下更精确控制上下文大小。

- **Prompt 级优化**：
  - 通过更明确的指令与输出约束，减少模型产生无效 `wait()` 或多余步骤，从源头上降低总体步数。

综合以上措施，本次优化在**不改变核心业务逻辑与交互模式**的前提下，针对会话历史管理与等待时间控制做了重点改进，并补充了统一的基准测试与调优文档，便于后续持续监控与迭代优化。
